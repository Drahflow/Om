Using

Om can be used as a library or a stand-alone interpreter.

Library

Om is a header-only C++ library that can be incorporated into any C++ or Objective-C++ project as follows:

  • Add the Om code directory to the include path and include the desired files. Inclusion of any operation header files will automatically add the corresponding operation to the global system. Include om.hpp to include all Om header files.
  • Configure the project to link to the required ICU (ICU4C 49.1.2 or greater) and Boost (1.53.0 or greater) library dependencies as necessary, and build these with the correct configuration for the project. See make.sh and make.bat for guidance.
  • Call the Om::System::Initialize function prior to use (e.g. in the main function), passing in the desired UTF-8 locale string (e.g. "en_US.UTF-8").
  • Construct an Om::Environment, populate with any additional operator-program mappings, and call one of its Om::Environment::Evaluate functions to evaluate a program.

Projects

Project files that build the interpreter, unit tests, and documentation are generated by CMake via provided scripts.

Targets

The generated projects contain the the following targets.

Om

The Om target builds the Om interpreter executable into products/projects/[platform]. The Om interpreter:

  • Accepts an optional command-line argument that specifies the desired UTF-8 locale string. The default value is "en_US.UTF-8".
  • Reads input from the standard input stream, ending at the first unbalanced end brace, and writes output to the standard output stream as it is computed.

Om.Documentation

The Om.Documentation target builds the Om documentation into the following folders in products/documentation/doxygen:

  • html: The HTML documentation. To view in a browser, open index.html.
  • xml: The XML documentation, which can be read by an integrated development environment to show context-sensitive documentation.

Om.Test

The Om.Test target builds the Om.Test test executable into products/projects/[platform], which runs all unit tests.

This target only exists if the scripts are supplied with the unit test library path.

Platforms

Xcode

Building on Xcode requires that Doxygen and Graphviz (via MacPorts) are installed and available to applications via the path variable. This article explains how to do so.

The following scripts have been tested with Xcode 4.6 on Mac OS X 10.8.2 (Mountain Lion):

  • premake.sh:
    • Builds the external library dependencies as required for Om.
    • Takes the following arguments (in order):
      • The ICU (ICU4C 49.1.2 or greater) root directory path. To create this directory, download and unpack the ICU4C 50.1.2 code archive from the ICU download page.
      • The Boost (1.53.0 or greater) root directory path. To create this directory, download and unpack the Boost 1.53.0 code archive from the Boost download page.
      • Optional: The UnitTest++ (1.4) root directory path. If omitted, unit tests will be excluded from the build. To create this directory, download and unpack the UnitTest++ 1.4 code archive from the UnitTest++ download page.
  • make.sh:
    • Creates the Om.xcodeproj file in products/projects/xcode.
    • Takes the same arguments as premake.sh.
  • remake.sh:
    • Updates the Xcode project to reflect changes to the code directory tree.
    • Takes no arguments.
  • unmake.sh:
    • Removes the directory generated by make.sh.
    • Takes no arguments.

Visual Studio

Building on Visual Studio requires that Doxygen and Graphviz are installed and in the path variable.

The following scripts have been tested with Visual Studio 2010 Express on Windows Vista:

  • premake.bat:
    • Builds the external library dependencies as required for Om.
    • Takes the following arguments (in order):
      • The ICU (ICU4C 49.1.2) root directory path. To create this directory, download and unpack the ICU4C 49.1.2 code archive from the ICU download page. (Note that ICU4C 50.1.1 or 50.1.2 cannot be used due to build issues on Windows.)
      • The Boost (1.53.0 or greater) root directory path. To create this directory, download and unpack the Boost 1.53.0 code archive from the Boost download page.
      • Optional: The UnitTest++ (1.4, patched with Visual Studio support as necessary) root directory path. If omitted, unit tests will be excluded from the build. To create this directory, get the UnitTest++ 1.4 code via Subversion checkout (described on the UnitTest++ developer page) and apply the patch attached to the Visual Studio 2010 support ticket. The UnitTest++ root directory is the UnitTest++ subdirectory of the Subversion checkout directory.
  • make.bat:
    • Creates the Om.sln file in products/projects/vs.
    • Takes the same arguments as premake.bat.
  • remake.bat:
    • Updates the Visual Studio solution to reflect changes to the code directory tree.
    • Takes no arguments.
  • unmake.bat:
    • Removes the directory generated by make.bat.
    • Takes no arguments.

Other

Because Om is written in portable C++, it should be possible to build the code on any mainstream platform by adapting the scripts and CMakeLists.txt file. Please share any enhancements to the build process and/or report any code portability issues encountered.
