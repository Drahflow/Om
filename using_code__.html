Om
A Simple Programming Language
Using the Code

The Om source code can be used for:

  • Building a stand-alone interpreter from a script-generated source project.
  • Including as a C++ header-only library.

Dependencies

The code depends on the following external libraries:

  • ICU (ICU4C 50.1.2 or greater)
  • Boost (1.53.0 or greater)

To run the scripts which build the above libraries and generate the project, the following programs are required:

  • CMake (2.8.10 or greater)
  • Cygwin (with bash, GNU make, ar, and ranlib) (Windows only)

To build the documentation target in the project, the following programs are required:

  • Doxygen (1.8.3.1 or greater)
  • Graphviz (2.30.1 or greater)

To ensure that the correct applications are used, the command line path should be in the following order:

  • Graphviz, Doxygen, and CMake
  • Cygwin ("[cygwin]/bin") (Windows only)
  • Any other paths

Project

Scripts are provided for generating a source project that contains targets for building the interpreter, unit tests, and documentation.

The "generate" script provides single-click generation of the project, including automatically downloading and building all external dependencies.

For more fine-grained control, the "internal" and "external" folders each contain respective "generate" scripts:

  • The internal generate script is responsible for generating the project file in "internal/output/artifacts/[platform]". Its arguments are passed directly to CMake which does all of the work of generating the project. The following additional arguments must be specified:
    • -D Icu:Path="[absolute path]" provides the absolute path of the ICU4C install directory.
    • -D Boost:Path="[absolute path]" provides the absolute path of the Boost install directory.
    • -D External:Path="[absolute path]" provides the absolute path to the "external" folder, where externals are generated automatically in the event that one of the above directories does not exist.
  • The external generate script is responsible for generating the external dependencies into "external/output/artifacts/[dependency]". The dependencies are downloaded and built as necessary.

Platforms

Xcode

Generate scripts ending in ".sh" are designed for use with Xcode. These scripts have been tested with Xcode 4.6 on Mac OS X 10.8.2 (Mountain Lion).

Visual Studio

Generate scripts ending in ".bat" are designed for use with Visual Studio 2010. These scripts have been tested with Visual Studio 2010 Express on Windows Vista.

Other

Because Om is implemented in a portable way, it should be possible to build the code on any mainstream platform through minor adaptation to the scripts and "CMakeLists.txt" files. Please share any enhancements to the build process and/or report any code portability issues encountered.

Targets

Om

The Om target builds the Om interpreter executable into "internal/output/artifacts/[platform]/[configuration]". See Using the Interpreter for instructions on how to use the interpreter.

Om.Test

The Om.Test target builds the Om.Test test executable, which runs all unit tests, into "internal/output/artifacts/[platform]/[configuration]".

Om.Documentation

The Om.Documentation target builds this documentation into the following folders in "internal/output/artifacts/Doxygen":

  • "html": The HTML documentation. To view in a browser, open "index.html".
  • "xml": The XML documentation, which can be read by an integrated development environment to show context-sensitive documentation.

Library

Om is a header-only C++ library that can be incorporated into any C++ or Objective-C++ project as follows:

  • Add the Om "internal/input/code" directory to the include path and include the desired files. Inclusion of any operation header files will automatically add the corresponding operation to the global system. Include "om.hpp" to include all Om header files.
  • Configure the project to link to the code dependencies as necessary, built with the correct configuration for the project. See "external/generate.sh" and/or "external/generate.bat" for guidance.
  • Call the Om::System::Initialize function prior to use (e.g. in the main function), passing in the desired UTF-8 locale string (e.g. "en_US.UTF-8").
  • Construct an Om::Environment, populate with any additional operator-program mappings, and call one of its Om::Environment::Evaluate functions to evaluate a program.

For more in-depth usage of the library, see the Om code documentation.
